## Story

**As a** developer do Yutubo,
**I want** implementar o backend Node.js com yt-dlp e fazer deploy no Oracle Cloud Always Free,
**so that** o frontend (componente DC) possa analisar URLs do YouTube e iniciar downloads reais de vídeo e áudio.

## Acceptance Criteria

1. Oracle Cloud ARM VM (Ubuntu 24.04) provisionada e acessível via SSH com IP público fixo.
2. Dependências instaladas na VM: Node.js 20.x, Python3, yt-dlp (via pip3), ffmpeg, Deno (obrigatório para assinaturas YouTube em 2026).
3. `POST /api/analyze` recebe uma URL do YouTube e retorna JSON com: título, canal, duração, lista de qualidades disponíveis (MP4: 2160p/1440p/1080p/720p/480p/360p; MP3: 320/256/192/128 kbps).
4. `POST /api/download` recebe URL + formato + qualidade + (opcional) cookies e inicia o download, fazendo streaming do arquivo diretamente para o cliente sem armazená-lo permanentemente no servidor.
5. `GET /api/progress/:id` retorna o progresso de um download em andamento (percentual, velocidade, tempo estimado) via Server-Sent Events (SSE) ou polling JSON.
6. O backend aceita upload de arquivo `cookies.txt` via `POST /api/cookies` para mitigar bloqueio de IP do YouTube em datacenters (necessário em 2026).
7. Download queue implementada: máximo 2 downloads simultâneos. Requests além do limite recebem resposta 429 com tempo estimado de espera.
8. nginx configurado como reverse proxy: porta 443 → backend 3000, com SSL via Let's Encrypt (certbot).
9. Serviço `yutubo-backend` configurado via systemd com auto-restart em falha.
10. CORS configurado para aceitar requests do domínio do frontend (Vercel/Cloudflare Pages).
11. Endpoint `GET /api/health` retorna `{ status: "ok", version: "1.0.0" }` para health checks.
12. Download de vídeo YouTube individual funciona end-to-end: URL → análise → seleção de qualidade → download no browser do usuário.

  server {
    listen 443 ssl;
    server_name api.seu-dominio.com;

### Problema Crítico: IP Blocking do YouTube (2026)

[Source: docs/deployment-options.md]

O YouTube bloqueia ativamente IPs de datacenters (incluindo Oracle Cloud) em 2026.

**Sintoma:**
```
ERROR: Sign in to confirm you're not a bot.
Use --cookies-from-browser or --cookies for authentication.
```

**Solução implementada nesta story:**
- Endpoint `POST /api/cookies` para upload de `cookies.txt` do usuário
- yt-dlp usa `--cookies /path/to/cookies.txt` ao fazer requests ao YouTube
- Usuário exporta cookies do browser com extensão "Get cookies.txt locally"

**Requisito crítico — Deno:**
yt-dlp em 2026 requer Deno para resolver assinaturas do YouTube. Sem Deno instalado, downloads falham com erro 403 mesmo com proxy.

### Estrutura de Comandos yt-dlp

```bash
# Análise de metadata
yt-dlp --dump-json "URL"

# Download MP4 1080p
yt-dlp -f "bestvideo[height<=1080][ext=mp4]+bestaudio[ext=m4a]/best[height<=1080]" \
  --merge-output-format mp4 \
  --cookies /path/to/cookies.txt \  # se disponível
  -o "/tmp/%(id)s.%(ext)s" \
  "URL"

# Download MP3 320kbps
yt-dlp -f "bestaudio" \
  --extract-audio --audio-format mp3 --audio-quality 0 \
  --cookies /path/to/cookies.txt \  # se disponível
  -o "/tmp/%(id)s.%(ext)s" \
  "URL"
```

### Configuração nginx para Streaming

`proxy_buffering off` é **obrigatório** para SSE (Server-Sent Events de progresso) e para streaming de arquivos grandes. Sem isso, nginx bufferiza o response e o progresso não chega ao cliente em tempo real.

`proxy_read_timeout 600s` — downloads de vídeos 4K em 1080p+ podem demorar vários minutos.

### Completion Notes List

**Scope delivered (local code — @dev):**

- Full Express backend under `backend/` implementing all 5 API endpoints
  (analyze, download, progress SSE, cookies upload, health).
- yt-dlp subprocess wrapper (`lib/ytdlp.js`) with quality mapping for MP4
  (2160p–360p via `height<=N`) and MP3 (320–128 kbps via `--audio-quality NK`),
  metadata normalization (video vs playlist), and progress-line parsing.
- Concurrency gate (`lib/queue.js`): max 2 simultaneous downloads, returns 429 +
  `Retry-After` with estimated wait when full (AC-7).
- In-memory progress registry (`lib/progress.js`) bridging the download route
  (writer) and the SSE route (reader), keyed by client `downloadId`.
- Streaming download: yt-dlp stdout piped to the HTTP response with
  `{ end: false }` so we control success vs. error termination; no permanent
  file is written. `Content-Disposition: attachment` + UTF-8 filename.
- Cookies upload via multer (`POST /api/cookies`) → `cookies/<uuid>.txt`,
  with size limit and a sanity check for Netscape cookie format (AC-6).
- Periodic cleanup sweeper (`lib/cleanup.js`) for stale `tmp/` and expired cookies.
- CORS configurable via `CORS_ORIGINS` env (AC-10). Health endpoint (AC-11).
- Deploy templates: `deploy/nginx.conf` (reverse proxy, `proxy_buffering off`
  for SSE/streaming, 600s timeouts, SSL via certbot) and
  `deploy/yutubo-backend.service` (systemd, auto-restart, env, hardening).
- Frontend `Descargador.dc.html`: `validate()` and `startDownload()` rewired to
  the real API. `validate()` calls `POST /api/analyze` and populates
  `state.video` / `state.playlist` / `state.items`. `startDownload()` generates
  a `downloadId`, opens an `EventSource` to `/api/progress/:id` for the live
  progress bar, and submits a hidden form POST to `/api/download` so the browser
  streams the attachment natively to disk. Added `api()` helper supporting
  same-origin and cross-origin (`window.YUTUBO_API` / `props.apiBase`) so the
  Vercel/Cloudflare frontend can target the Oracle backend.
- `renderVals()` now reads `video`/`playlist` from state (was hardcoded);
  hardcoded `videoData`/`playlistData` removed.

**Security (addresses PO Should-Fix SSRF/rate-limit):**

- `lib/validate.js` enforces an http(s) + YouTube-host allowlist before any URL
  reaches yt-dlp (SSRF guard). yt-dlp is always invoked with an argv array
  (never a shell string) — no shell injection possible.
- Cookie path resolution validates the id against a uuid regex and a
  `startsWith(cookiesDir)` check (path-traversal guard).
- Concurrency gate provides basic abuse protection (max 2 in-flight).

**Self-critique (Step 5.5/6.5):** Caught and fixed a streaming race condition —
`stdout.pipe(res)` auto-ended the response before the `close` handler could send
a JSON error on early yt-dlp failure. Fixed with `{ end: false }` + `bytesSent`
tracking so an early failure returns a clean 502 and a mid-stream failure
destroys the socket to signal truncation.
